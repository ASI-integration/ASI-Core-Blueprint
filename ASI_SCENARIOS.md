# CONTOUR 3 — OPERATIONS (CLEANING & TURNOVER)

## Purpose
Управление операционным состоянием объекта между заездами.
Контур отвечает за проверку, уборку, готовность объекта и обработку сбоев.

---

## Object States

Объект всегда находится ровно в одном состоянии:

- READY
- OCCUPIED
- INSPECTION_REQUIRED
- NEEDS_CLEANING
- CLEANING_IN_PROGRESS
- HOLD / INCIDENT

Состояние объекта ≠ статус брони.

---

## State Transitions

### Arrival
READY → OCCUPIED  
Триггер: CONFIRMED_STAY + доступ выдан  
Кто: система

---

### Departure (плановый или ранний)
OCCUPIED → INSPECTION_REQUIRED  
Триггер: выезд / окончание брони / ранний выезд  
Кто: система / оператор  
Метка события (опционально): EARLY_DEPARTURE

---

### Post-Stay Inspection
INSPECTION_REQUIRED → READY  
Условие: отклонений от стандарта не выявлено  
Кто: горничная / оператор / владелец

INSPECTION_REQUIRED → NEEDS_CLEANING  
Условие: выявлено хотя бы одно отклонение  
Кто: горничная / оператор

Проверка обязательна всегда, даже если гость «не жил».

---

### Cleaning
NEEDS_CLEANING → CLEANING_IN_PROGRESS  
Триггер: подтверждённый старт уборки  
Кто: горничная / оператор

CLEANING_IN_PROGRESS → READY  
Триггер: подтверждённое завершение уборки  
Кто: горничная (факт) / оператор (верификация)

Без подтверждения завершения объект не может стать READY.

---

### Incidents (Priority Path)
* → HOLD / INCIDENT  
Триггер: риск, авария, неопределённость  
Кто: система / оператор / владелец

HOLD / INCIDENT → INSPECTION_REQUIRED | NEEDS_CLEANING | READY  
Триггер: разрешение инцидента  
Кто: оператор / владелец

HOLD имеет приоритет над всеми состояниями.

---

## Cleaning Rules

- Уборка запускается только из NEEDS_CLEANING
- Уборка является следствием проверки, а не факта выезда
- Система не запускает и не завершает уборку автоматически
- Частичная уборка как состояние не допускается

---

## Failures & Exceptions

### Cleaner No-Show
- объект остаётся в NEEDS_CLEANING
- фиксируется FAILURE: NO_SHOW_CLEANER
- решение принимает оператор / владелец

### Cleaning Interrupted
- объект → HOLD / INCIDENT
- требуется повторная проверка
- уборка считается прерванной

### Emergency Replacement
- новая горничная начинает с проверки
- уборка считается новым циклом

### Loss of Confirmation
- без подтверждения завершения объект не может стать READY
- объект переводится в HOLD до верификации

---

## Core Canons

- Проверка обязательна всегда
- Уборка условна, не автоматична
- Без подтверждений нет READY
- Безопасность приоритетнее скорости

## ARCH FREEZE — Contour 3 (Operations: Cleaning & Turnover)

### Scope Closure
Contour 3 считается операционно завершённым при выполнении следующих условий:

- логика состояний объекта полностью определена
- все допустимые переходы между состояниями зафиксированы
- обязательная post-stay проверка формализована
- запуск уборки допускается только после проверки
- уборка требует подтверждённого начала и завершения
- сбои, прерывания и замены обработаны как отдельные сценарии
- состояние HOLD / INCIDENT имеет приоритет над всеми операциями

---

### System Responsibility Boundary
Система в рамках Contour 3 отвечает ТОЛЬКО за:

- фиксацию операционного состояния объекта
- блокировку недопустимых переходов
- удержание объекта в безопасном состоянии при неопределённости
- передачу управления человеку при сбоях

Система НЕ отвечает за:
- качество фактической уборки
- принятие экстренных решений без человека
- автоматическое завершение процессов

---

### Human Responsibility Boundary
Любые решения, выходящие за рамки канонических переходов,
принимаются человеком (оператор / владелец).

---

### Explicit Exclusions
Contour 3 НЕ включает и НЕ затрагивает:
- Arrival Outcome
- Booking logic
- PRC (Pre-Arrival Communication)
- финансовые расчёты
- коммуникации с гостями или площадками
- UI / API / автоматизацию исполнения

---

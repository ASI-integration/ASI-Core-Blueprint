\# AI\_PROJECT\_MAP (ASI) — сборка без Prisma



Цель: иметь "source of truth" (логическое ядро) отдельно от кода, чтобы любой агент/инструмент мог собрать backend/frontend по правилам, без привязки к Prisma и без того, чтобы проект разваливался из-за ORM.



---



\## 0) Как мы работаем (коротко)



Есть 3 слоя:



1\) \*\*00\_CORE\*\* — неизменяемые правила (каноны): как мы вообще думаем, как принимаем решения, что считается “правильным”.

2\) \*\*01\_DOMAIN\*\* — доменная логика (контуры): сущности, сценарии, состояния, события, инварианты.

3\) \*\*02\_RUNTIME\*\* — текущий контекст выполнения: что уже сделано, что собираем сейчас, что сломано, что в приоритете.



Код (backend/frontend) — это “реализация”, которую можно пересобирать сколько угодно раз, пока ядро остаётся стабильным.

\## 1) Папки и “истина”



\### 00\_CORE (правила игры)

Содержимое (пример по твоей папке):

\- любые агенты должны ссылаться на CORE как на закон.

\### 01\_DOMAIN (что строим)

Содержимое (контуры + сценарии):

Назначение:

\### 02\_RUNTIME (в каком мы состоянии)

Содержимое:

Назначение:

\- “живой” слой, меняется часто.



---



\## 2) Важная мысль про Prisma (и почему она ломает всё)



Prisma — это implementation detail (деталь реализации).

Когда Prisma становится “позвоночником” домена (внедрена во все сервисы/модули/контроллеры),

любой сбой генерации клиента превращает проект в тыкву.



Нам нужно:

\- \*\*Domain layer не должен знать, что такое Prisma.\*\*

\- В runtime-коде DB доступ — через \*\*порт/интерфейс\*\* (Repository / Gateway), который можно реализовать:

&nbsp; - через SQL (Kysely/pg)

&nbsp; - через Knex

&nbsp; - через TypeORM

&nbsp; - через что угодно

но ДОМЕНУ всё равно.



---



\## 3) “Как делают PMS” (в двух словах)



Нормальные PMS держат границы:

\- домен отдельно

\- хранение данных отдельно

\- интеграции отдельно



ORM (если есть) — инструмент, но не фундамент.



---



\## 4) Целевой дизайн без Prisma (минимально)



Backend (NestJS) разделяем так:



\- src/domain/\*\*        ← чистая логика (без БД, без Nest-специфики по возможности)

\- src/app/\*\*           ← use-cases / services (координация, транзакции, вызов репозиториев)

\- src/infra/db/\*\*      ← реализация репозиториев на SQL (kysely/pg) + миграции

\- src/api/\*\*           ← controllers / dto / validation

\- src/integrations/\*\*  ← OTA, webhooks, внешние сервисы



Ключ: Controllers → App services → Repositories (порт) → DB (реализация)



---



\## 5) План миграции “Prisma -> No Prisma” (без боли)



\### Этап A — стабилизировать сборку

\- убрать импорты PrismaModule/PrismaService/PrismaClient из модулей, сервисов, тестов

\- удалить/заменить prisma/\*\* пакет в src (или вынести в архив)



\### Этап B — дать проекту DB слой без Prisma

\- ставим: pg + kysely

\- делаем DbModule (provider для Kysely)

\- создаём 2-3 репозитория минимум (Units, Bookings, WebhookEvents) — только то, что нужно для запуска



\### Этап C — постепенно переносить остальные модули

\- по одному модулю за раз, без “массовых” замен регулярками

\## 7) Как этим пользоваться (коротко)



Когда даёшь задачу агенту (Gemini/Opus/другому), прикладывай:

\- 00\_CORE целиком

\- нужные контуры из 01\_DOMAIN

\- ASI\_CONTEXT + PROJECT\_STATE из 02\_RUNTIME



И формулируй запрос так:

“Собери backend в соответствии с CORE+DOMAIN, Prisma запрещена, DB через Kysely/pg, сделать минимальный рабочий запуск + 1-2 эндпоинта.”



---



\## 8) “Почему так делают неправильно” (твоя главная боль)



Потому что:

\- людям (и агентам) проще “быстро подключить ORM и пошло-поехало”

\- потом домен незаметно срастается с БД

\- и любая поломка генерации/клиента ломает всю систему



Мы делаем наоборот: домен — железный, реализация — сменная.



---



\## 9) Следующие 2 шага (чтобы не разносить всё сразу)



Шаг 1: фиксируем ядро

\- утвердить, что 00\_CORE/01\_DOMAIN/02\_RUNTIME — это “не трогаем без причины”.

\- код можно выбрасывать и пересобирать — ядро остаётся.



Шаг 2: собираем “чистый backend skeleton без Prisma”

\- NestJS + DbModule (Kysely/pg)

\- UnitsModule с 2 методами (list/get) как proof-of-life

\- после этого можно спокойно наращивать.
